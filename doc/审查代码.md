# 审查代码

这个清单用于审查当前 Python 主线的改动，重点不是样式，而是行为风险。

## 1. 入口层

关注文件：

- `pinchRatioClustering.py`
- `genSimMatrix.py`
- `prc.py`
- `pyprc/cli_pinch_ratio_clustering.py`
- `pyprc/cli_gen_sim_matrix.py`

检查点：

- 命令行入口是否仍能调用核心流程
- 兼容导入入口是否未断裂
- 包入口和脚本入口是否行为一致

## 2. 参数解析层

关注文件：

- `pyprc/core.py`

检查点：

- 参数默认值是否被意外修改
- 历史兼容参数名是否仍可识别
- 参数越界时是否有明确报错

## 3. 数据和构图层

检查点：

- 点数据读取是否正确处理标签列
- 邻接输入是否正确处理 `adjNodeOffset`
- KNN/Gaussian 的自动参数路径是否还能触发
- 稀疏和稠密路径是否在常见输入上保持一致

## 4. TILO 和 PRC 核心

检查点：

- `calcBoundaries` 的公开签名和内部签名是否都可用
- `TILO`/`RefineTILO` 的兼容调用是否未断
- 递归切分后顺序回写是否正确
- `policyPRCMetric` 是否只影响选点，不破坏无关路径

## 5. 输出层

检查点：

- 标签顺序是否与原始样本顺序一致
- `saveOrder`、`saveLabels`、`saveMultipleRuns` 是否各自独立生效
- 返回结构是否完整

## 6. 文档同步

检查点：

- `readme.rst` 是否仍与当前仓库一致
- `doc/` 是否还引用已删除的 C++ 文件或旧目录
- 示例命令是否还能直接运行

## 7. 最小回归集

建议每次改动后至少执行：

1. `tests/test_pyprc_smoke.py`
2. `tests/d1.txt` 的 CLI 邻接图用例
3. `tests/iris_all.txt` 的点数据用例

## 8. 审查时应避免的错误结论

以下结论不能因为“样例跑通”就直接下：

- 与历史 C++ 完全等价
- 所有随机初始化路径都已验证
- 所有实验性参数都可靠

如果没有对照测试，就应该明确写成“当前已验证主流程可运行”。
