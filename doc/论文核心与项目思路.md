# 论文核心与项目思路

## 这个仓库解决什么问题

PRC/TILO 处理的是图聚类问题。

它的基本路线不是直接在嵌入空间里做几何切分，而是：

1. 先把数据表示成加权图
2. 再给图顶点寻找一个“好”的线性顺序
3. 用顺序上的边界宽度变化识别可切分结构
4. 用指标选择切分点，递归得到多个簇

## 两篇论文的分工

### 1. thin decomposition / thin position 方向

这部分给出的是理论直觉：

- 顶点顺序可以诱导一条边界序列
- 顺序不同，边界序列的“宽度”不同
- 通过局部 shift 可以持续改进顺序
- 在某些局部极小位置会出现可解释的簇边界

### 2. Pinch Ratio Clustering

这部分把理论方法变成可执行聚类流程：

- 先运行 TILO 得到更稳定的顺序
- 再评估候选切分点
- 通过 PinchRatio、NCut 等指标选点
- 递归切分直到达到目标簇数

## 当前 Python 实现如何对应论文

当前仓库里的对应关系如下：

- 图读取与点数据构图：`readData`、`gaussSimMatrix`、`knnSimMatrix`
- 顶点顺序对象：`OrderObject`
- 边界序列对象：`BoundaryObject`
- TILO 主过程：`TILO`、`RefineTILO`
- PRC 切分：`findSplitLocation`、`pinchRatioClustering`

## 算法直觉

可以把整个过程理解为两步：

### 第一步：先把图排顺

如果一个顺序把“应该相邻”的点尽量放在一起，那么前缀边界宽度会更集中，局部结构会更明显。

### 第二步：再从顺序里找切点

当顺序已经较稳定时，某些切分位置会比其他位置更自然。PRC 用指标去评价这些位置，从而决定在哪里分簇。

## 当前仓库的实现取舍

这次整理后的仓库做了两个工程取舍：

1. 优先保留论文主流程
2. 不再保留旧 C++ 工程形态作为用户入口

因此，当前仓库重点是“可运行的算法实现”，不是“保留所有历史工程文件”。

## 应该怎样理解“与论文一致”

合理的表述应该是：

- 当前代码遵循论文主流程和主要结构
- 当前代码可以运行主要实验链路
- 当前代码尚未完成对历史实现的全量结果对照

这比直接宣称“已经完全等价”更严谨。
