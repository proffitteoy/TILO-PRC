# 架构重构与接口分层说明（2026-05-17）

本文记录本次在纯 Python 主线下做的“接口先行”重构，目标是在不改核心数学流程的前提下，把实验协议层、编排层和算法层解耦，为后续接入其他后端实现预留稳定接口。

## 1. 重构目标

- 保持 `pyprc.algorithm` 的 TILO/PRC 计算逻辑不变
- 把构图策略、policy 克隆、best-of-trials 挑选等“流程编排”逻辑抽离
- 为未来替换执行后端（例如重新接 C++、GPU 或远程服务）预留协议接口
- 避免 `experiments/*` 与 `pyprc/cli.py` 各自复制一套同类流程代码

## 2. 新增模块

- 新增 `pyprc/pipeline.py`

核心内容：

- `PrcExecutionBackend`（Protocol）
  - `init_random_order(n)`
  - `run_pinch_ratio_clustering(matrix, order, num_partitions, policy)`
- `PythonPrcExecutionBackend`
  - 当前默认后端，直接调用纯 Python `pinchRatioClustering_storage`
- `GraphBuildConfig`
  - 统一点数据构图配置（gauss/knn、sparse/dense、阈值与参数）
- `build_similarity_matrix_from_points`
  - 统一构图入口，返回 `MatrixStorage + graph meta`
- `clone_prc_policy`
  - 统一策略克隆与覆写逻辑
- `run_best_of_trials`
  - 统一“多次随机初始化 + lexicographic boundary 选优”流程
- `orders_from_label_blocks`
  - 统一 label -> 初始 order 候选生成
- `run_from_candidate_orders`
  - 统一“候选初始顺序集合挑优 + 可选失败回退”流程

## 3. 调整范围

- `experiments/reproduce_paper_fig_ari.py`
  - 对外函数签名保持不变
  - 内部改为委托 `pyprc.pipeline`，减少脚本内重复实现
- `pyprc/cli.py`
  - 点数据构图分支改为复用 `build_similarity_matrix_from_points`
- `pyprc/__init__.py`
  - 导出新的服务层 API，供外部直接调用

## 4. 分层边界（当前）

- 算法层：`pyprc/algorithm.py`
  - 负责 TILO、RefineTILO、split、PRC 递归切分
- 编排层：`pyprc/pipeline.py`
  - 负责策略装配、构图、多次运行挑优、候选 order 执行
- 入口层：`pyprc/cli.py` 与 `experiments/*`
  - 负责参数解析、实验协议和结果呈现

## 5. 后续建议

- 若要恢复“多后端对齐验证”，只需新增 backend 实现并注入到 `run_best_of_trials/run_from_candidate_orders`
- 若要继续拆大脚本，可优先把 `experiments/reproduce_paper_fig_ari.py` 的 baseline 搜索与绘图进一步拆成独立模块

## 6. 类化编排（本次新增）

- `pyprc.pipeline.PrcPipelineService`
  - 将原先分散的函数方法统一收口为类方法
  - 支持通过构造参数注入 `backend`
- `pyprc.pipeline.DEFAULT_PIPELINE`
  - 默认服务实例，供函数兼容层复用
- 兼容策略
  - 旧函数名（如 `run_best_of_trials`）仍保留，对外行为不变
  - 旧函数内部转调 `DEFAULT_PIPELINE`，便于逐步迁移调用方到类接口
